# Prompt optimization for plan-execute-judge

An analysis of applying automatic prompt optimization — GEPA and MIPROv2 — to
this pipeline's prompts, and the design of the harness in `optimize/` +
`evals/` that implements the GEPA path. Written 2026-07; optimizer APIs
verified against gepa 0.1.3.

## The optimization surface

The pipeline's behavior is governed by exactly five prompt templates, one per
LLM phase, now externalized in `src/prompts.ts` (`DEFAULT_PROMPTS`):

| template | phase | dynamic input (via `{{INPUT_DATA}}`) |
|---|---|---|
| `research` | optional deep research | task, sources, notes, agent index/count (+ `{{SOURCE_ACCESS}}`, backend-specific) |
| `plan` | planning | task, research brief, agent index/count |
| `refinements` | merge candidate plans | task, research, candidate plans |
| `execute` | implementation | plan, prior judge gaps, pipeline files |
| `judge` | review + typed verdict | task, plan, review commands, baseline ref |

There are no system prompts; each template is the entire user prompt of a
fresh agent run. `PEJ_PROMPTS_FILE` overrides any subset of them as data
(JSON `{phase: template}`), which is the injection point every optimizer uses.
The judge's output shape is enforced by JSON schema (`VerdictSchema`), not
prompt text, so optimizing the judge template cannot break verdict parsing.

## Metric design: avoiding judge circularity

The obvious metric — `verdict.pass` — is generated by an LLM reading one of
the very prompts we might optimize. Optimizing plan/execute against the judge
alone would reward *convincing the judge*; optimizing the judge against itself
is meaningless. The harness therefore scores each rollout against a **hidden
acceptance check** (`evals/checks/*.mjs`): a deterministic script, invisible
to the pipeline, that verifies the task's actual behavior in the worktree and
re-runs the fixture's test suite. Score per rollout (`optimize/pej.py`):

```text
0.70 · hidden check passed
0.15 · judge verdict agrees with the hidden check   (honest judging, either direction)
0.15 · hidden check passed AND judge passed it in round 1   (efficiency shaping)
```

Ground truth dominates; the judge only earns points for *agreeing with
reality*, which stays meaningful even if the judge template itself is later
optimized. Crashes/timeouts score 0 with the failure noted in feedback.

**Optimizing the judge itself** (deferred): needs its own labeled dataset —
(worktree state, plan, task) → known-correct verdict — built by seeding
fixtures with known-good and known-broken implementations (the hidden checks
already define "broken"), then scoring the judge on verdict agreement and gap
usefulness. The `PipelinePhases` seam in `src/orchestrator.ts` lets `runJudge`
be driven in isolation for this.

## Why GEPA (and why not MIPROv2)

The binding constraint: **one metric call = one full pipeline run** — several
agentic phases, minutes of wall clock, real token cost. That changes which
optimizer is economical.

**GEPA** (Genetic-Pareto, [paper](https://arxiv.org/abs/2507.19457), ICLR 2026;
`pip install gepa`) fits this system unusually well:

- **Sample efficiency.** GEPA mutates prompts by *reading* execution traces
  and textual feedback with a reflection LM, instead of inferring gradients
  from scalar scores over many trials. The paper reports outperforming GRPO
  by ~10% on average with up to 35x fewer rollouts, and beating MIPROv2 by
  over 10% on aggregate. Budgets of 100–200 metric calls are useful; MIPROv2
  wants high hundreds to thousands.
- **Native fit for rich feedback.** This pipeline already produces exactly
  what GEPA reflects on: `verdict.gaps[]` (typed, per-requirement failure
  descriptions), hidden-check expected-vs-actual output, per-round verdicts,
  diffstats, crash notes. A scalar-only optimizer throws all of that away.
- **Multi-component candidates.** A candidate is `{plan: ..., execute: ...}`;
  GEPA mutates one component per iteration (round-robin) and keeps a Pareto
  frontier over tasks, so a plan-prompt that fixes kvstore tasks doesn't get
  discarded for losing an argsy task.
- **Adapter, not framework.** The standalone `GEPAAdapter` protocol
  (`evaluate(batch, candidate, capture_traces)` → scores + trajectories;
  `make_reflective_dataset(...)` → feedback records) wraps an *external*
  system directly — `optimize/adapter.py` shells out to `node dist/index.js`
  per rollout. No re-modeling of the pipeline inside an LLM framework.

**MIPROv2** (DSPy) is documented here as analysis, deliberately not built:

- Its two levers are bootstrapped **few-shot demos** and Bayesian (TPE)
  search over **instruction candidates**, optimized jointly. Demos don't map
  to this system: a "demonstration" of the execute phase would be an entire
  multi-turn tool-use transcript, far too large to inline and not something
  the Agent SDK prompt format supports. So MIPROv2 degrades to 0-shot
  instruction search (`max_bootstrapped_demos=0, max_labeled_demos=0`).
- The Bayesian loop needs many trials to be better than random search:
  with ~30 trials × ~25-task minibatches you're at ~750 pipeline runs
  *before* full evaluations — at even $1–3/rollout that's $1–2k+ per
  optimizer run, vs GEPA's low hundreds of dollars ceiling at the same
  rollout cost (and low tens at haiku prices).
- It consumes only a scalar metric, discarding the judge's gap text — the
  richest signal this pipeline produces.
- To use it at all, each phase must be wrapped as a `dspy.Module` whose
  `forward()` shells out to the pipeline and whose signature instruction is
  the template — feasible (the sketch mirrors `optimize/adapter.py`) but
  every MIPROv2-specific benefit (demos, grounding-aware proposal from
  program code) is inert on an opaque external program.
- TypeScript-native alternative: Ax (`ax-llm/ax`) ships MiPRO, but its v2
  now requires a Python optimizer service anyway, and its GEPA would require
  re-expressing phases as Ax programs. Python-with-subprocess is the smaller
  integration.

If a MIPROv2 comparison is ever wanted: reuse `pej.run_rollouts` as the DSPy
metric, wrap one phase at a time, run 0-shot with `auto="light"` and a
minibatch of ≤4 tasks, and expect it to be noise-limited at that budget.

## Architecture of the harness

```text
evals/
  fixtures/{argsy,slugger,kvstore}/   3 dependency-free Node mini-projects
  tasks/<id>.json                     12 tasks (8 train / 4 val), natural language only
  checks/<id>.mjs                     hidden ground-truth checks (never seen by the pipeline)
optimize/
  pej.py         rollout runner: fixture -> temp git repo -> node dist/index.js
                 (PEJ_PROMPTS_FILE, PEJ_RESULT_FILE, PEJ_MAX_ROUNDS=2) -> hidden
                 check -> score + feedback text
  adapter.py     GEPAAdapter; also rejects candidates that lost {{INPUT_DATA}}
                 without spending rollouts
  baseline.py    scores defaults or a candidate file (--prompts) on any split
  run_gepa.py    gepa.optimize(...) driver; reflection via `claude -p` (login
                 auth) or --reflection-lm <litellm-id>; resumable --run-dir
src/prompts.ts   externalized templates + PEJ_PROMPTS_FILE loader (validated)
PEJ_RESULT_FILE  machine-readable run summary incl. every round's verdict
```

Cost controls: rollouts default to `claude-haiku-4-5-20251001` at `medium`
effort, 2 max rounds, ≤3 parallel workers; a 150-call run is low tens of
dollars. The reflection LM should stay opus-class — it runs once per
iteration, not per rollout. **Transfer risk:** prompts tuned at haiku may not
transfer to the production model; always re-validate the winner with
`baseline.py --prompts ... --model claude-opus-4-8 --effort high --split val`,
and treat train/val separation as sacred (val tasks are never reflected on).

## Extending to the remaining prompts

- **`refinements`**: needs multi-plan rollouts — run with `PEJ_PLAN_AGENTS=2+`
  (wire it into `RolloutConfig`), roughly doubling plan-phase cost per rollout.
  Add `refinements` to `--components` once that's in place.
- **`research`**: needs tasks with research sources and `{{SOURCE_ACCESS}}`
  preserved through mutation (extend `validate_prompt_override`); rollouts get
  slower and web-dependent, so keep a dedicated task subset.
- **`judge`**: rejected by `run_gepa.py` — see the circularity section. Build
  the labeled-verdict dataset first, and never let the judge being optimized
  contribute the agreement or round-count terms of its own score.

Known limits of the current benchmark: 12 tasks on 3 toy fixtures is enough to
prove the loop and catch gross prompt regressions, not to certify general
improvement — treat val-split gains under ~0.1 mean score as noise, and grow
the task set (more fixtures, one larger realistic repo) before trusting bigger
claims.
